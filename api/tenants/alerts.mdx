## Overview

Alerts are generated when sensor readings exceed thresholds, sensors go offline, or actuators fail. Each alert belongs to a tenant and can be assigned different severity levels.

## Get All Alerts for Tenant

<RequestExample>
```bash GET /api/v1/tenants/{tenantId}/alerts
curl -X GET "https://api.invernaderos.com/api/v1/tenants/1/alerts" \
  -H "Authorization: Bearer YOUR_TOKEN"
```
</RequestExample>

Retrieve all alerts belonging to a specific tenant.

### Path Parameters

<ParamField path="tenantId" type="long" required>
  Unique identifier of the tenant
</ParamField>

### Response

<ResponseField name="id" type="long">
  Unique identifier of the alert
</ResponseField>

<ResponseField name="code" type="string">
  Unique readable code (e.g., "ALT-00001")
</ResponseField>

<ResponseField name="tenantId" type="long">
  ID of the tenant that owns this alert
</ResponseField>

<ResponseField name="sectorId" type="long">
  ID of the sector where the alert occurred
</ResponseField>

<ResponseField name="sectorCode" type="string">
  Code of the sector (e.g., "SEC-00001")
</ResponseField>

<ResponseField name="alertTypeId" type="short">
  ID of the alert type (1=THRESHOLD_EXCEEDED, 2=SENSOR_OFFLINE, 3=ACTUATOR_FAILURE, 4=SYSTEM_ERROR)
</ResponseField>

<ResponseField name="alertTypeName" type="string">
  Name of the alert type
</ResponseField>

<ResponseField name="severityId" type="short">
  ID of severity (1=INFO, 2=WARNING, 3=ERROR, 4=CRITICAL)
</ResponseField>

<ResponseField name="severityName" type="string">
  Name of severity level
</ResponseField>

<ResponseField name="severityLevel" type="short">
  Numeric severity level (1-4)
</ResponseField>

<ResponseField name="message" type="string">
  Short descriptive message
</ResponseField>

<ResponseField name="description" type="string">
  Detailed description of the alert
</ResponseField>

<ResponseField name="isResolved" type="boolean">
  Whether the alert has been resolved
</ResponseField>

<ResponseField name="resolvedAt" type="timestamp">
  When the alert was resolved (ISO 8601)
</ResponseField>

<ResponseField name="resolvedByUserId" type="long">
  ID of the user who resolved the alert
</ResponseField>

<ResponseField name="resolvedByUserName" type="string">
  Username of the user who resolved the alert
</ResponseField>

<ResponseField name="createdAt" type="timestamp">
  Creation timestamp (ISO 8601)
</ResponseField>

<ResponseField name="updatedAt" type="timestamp">
  Last update timestamp (ISO 8601)
</ResponseField>

## Create Alert

  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \

## Update Alert

  -H "Content-Type: application/json" \

    "severityId": 2,

## Alert Filtering

You can filter alerts using query parameters (implementation may vary):

### Filter by Resolution Status

<Tabs>
  <Tab title="Unresolved Alerts">
    ```bash
    GET /api/v1/tenants/1/alerts?isResolved=false
    ```
    
    Returns only unresolved alerts that require attention.
  </Tab>
  <Tab title="Resolved Alerts">
    ```bash
    GET /api/v1/tenants/1/alerts?isResolved=true
    ```
    
    Returns alerts that have been resolved.
  </Tab>
  <Tab title="Critical Alerts">
    ```bash
    GET /api/v1/tenants/1/alerts?severityId=4
    ```
    
    Returns only CRITICAL severity alerts.
  </Tab>
</Tabs>

### Filter by Alert Type

- **THRESHOLD_EXCEEDED** (alertTypeId=1): Sensor readings exceed configured thresholds
- **SENSOR_OFFLINE** (alertTypeId=2): Sensor stopped sending data
- **ACTUATOR_FAILURE** (alertTypeId=3): Actuator malfunction detected
- **SYSTEM_ERROR** (alertTypeId=4): System-level errors

### Filter by Severity

<Tabs>
  <Tab title="INFO (1)">
    Informational messages, no action required.
  </Tab>
  <Tab title="WARNING (2)">
    Potential issues that should be monitored.
  </Tab>
  <Tab title="ERROR (3)">
    Errors that require attention but are not critical.
  </Tab>
  <Tab title="CRITICAL (4)">
    Urgent issues requiring immediate action.
  </Tab>
</Tabs>
