## Overview

The Device Catalog provides centralized management for:

- **Device Categories**: SENSOR or ACTUATOR classifications
- **Device Types**: Specific sensor/actuator types (temperature, humidity, pumps, valves, etc.)
- **Actuator States**: Operational states for actuators (ON, OFF, AUTO, MANUAL, ERROR)

This catalog enables dynamic device configuration and type-safe device management across greenhouses.

<Warning>
  **Deletion Constraints**
  
  Cannot delete device types that have associated devices (sensors or actuators). You must:

  2. Use deactivation (`PATCH /deactivate`) instead for soft delete
</Warning>

## Control Types Explained

<AccordionGroup>
  <Accordion title="BINARY Control">
    **Use Case**: Simple on/off actuators
    
    **Examples**:
    - Irrigation pumps (ON/OFF)
    - Grow lights (ON/OFF)
    - Heaters (ON/OFF)
    
    **Data Type**: BOOLEAN
    
    **States**: ON, OFF
    
    **Implementation**:
    ```json
    {
      "name": "IRRIGATION_PUMP",
      "categoryId": 2,
      "controlType": "BINARY",
      "dataType": "BOOLEAN"
    }
    ```
  </Accordion>
  
  <Accordion title="CONTINUOUS Control">
    **Use Case**: Variable speed/position actuators
    
    **Examples**:
    - Vent windows (0-100% open)
    - Variable speed fans (0-100% speed)
    - Dimmers (0-100% brightness)
    
    **Data Type**: INTEGER or DECIMAL
    
    **Range**: 0-100 (percentage)
    
    **Implementation**:
    ```json
    {
      "name": "VENT_WINDOW",
      "categoryId": 2,
      "controlType": "CONTINUOUS",
      "dataType": "INTEGER",
      "minExpectedValue": 0,
      "maxExpectedValue": 100,
      "defaultUnitId": 2  // % symbol
    }
    ```
  </Accordion>
  
  <Accordion title="MULTI_STATE Control">
    **Use Case**: Actuators with predefined states
    
    **Examples**:
    - Irrigation valves (CLOSED, ZONE_1, ZONE_2, ZONE_3)
    - Multi-speed fans (OFF, LOW, MEDIUM, HIGH)
    - Heating systems (OFF, ECO, COMFORT, BOOST)
    
    **Data Type**: TEXT or INTEGER (state index)
    
    **States**: Defined in actuator_states catalog
    
    **Implementation**:
    ```json
    {
      "name": "IRRIGATION_VALVE",
      "categoryId": 2,
      "controlType": "MULTI_STATE",
      "dataType": "TEXT"
    }
    ```
  </Accordion>
</AccordionGroup>

// Generate device configuration form based on device type
const loadDeviceTypeConfig = async (deviceTypeId) => {
  const deviceType = await fetch(
    `https://api.invernaderos.com/api/v1/catalog/device-types/${deviceTypeId}`,

          options: states.map(s => ({ value: s.id, label: s.name }))
        }
      };
    
    default:
      return { component: 'Input', props: { label: deviceType.name } };
  }

```python Sensor Value Validation
import requests

def validate_sensor_reading(device_type_id, value):
    """Validate sensor reading against device type constraints"""
    
    # Get device type configuration
    response = requests.get(
        f"https://api.invernaderos.com/api/v1/catalog/device-types/{device_type_id}",
        headers={"Authorization": f"Bearer {token}"}
    )
    device_type = response.json()
    
    # Check if value is within expected range
    if device_type["minExpectedValue"] is not None:
        if value < device_type["minExpectedValue"]:
            return {
                "valid": False,
                "error": f"Value {value} below minimum {device_type['minExpectedValue']}"
            }
    
    if device_type["maxExpectedValue"] is not None:
        if value > device_type["maxExpectedValue"]:
            return {
                "valid": False,
                "error": f"Value {value} exceeds maximum {device_type['maxExpectedValue']}"
            }
    
    return {"valid": True}

# Usage
result = validate_sensor_reading(1, 25.5)  # Temperature: 25.5°C
if not result["valid"]:
    print(f"Invalid reading: {result['error']}")
```

```kotlin Actuator State Management (Android)
class ActuatorStateManager(private val api: InvernaderosApi) {
    
    // Cache actuator states for offline use
    private val stateCache = mutableMapOf<Short, ActuatorStateResponse>()
    
    suspend fun cacheActuatorStates() {
        val states = api.getActuatorStates()
        states.forEach { state ->
            stateCache[state.id] = state
        }
    }
    
    // Get color for state visualization
    fun getStateColor(stateId: Short): Int {
        val state = stateCache[stateId] ?: return Color.GRAY
        return try {
            Color.parseColor(state.color)
        } catch (e: Exception) {
            Color.GRAY
        }
    }
    
    // Check if actuator is operational
    fun isOperational(stateId: Short): Boolean {
        return stateCache[stateId]?.isOperational ?: false
    }
    
    // Get all operational states for UI dropdown
    suspend fun getOperationalStates(): List<ActuatorStateResponse> {
        return api.getOperationalActuatorStates()
    }
}

// Usage in UI
val stateManager = ActuatorStateManager(api)
stateManager.cacheActuatorStates()

// Display actuator status with color
val stateColor = stateManager.getStateColor(actuator.currentStateId)
actorStatusView.setBackgroundColor(stateColor)

// Check if actuator needs attention
if (!stateManager.isOperational(actuator.currentStateId)) {
    showMaintenanceAlert(actuator)
}
```

## Best Practices

<AccordionGroup>
  <Accordion title="Sensor Types">
    When creating sensor types:
    
    1. **Set realistic value ranges** (`minExpectedValue`, `maxExpectedValue`) for validation
    2. **Choose appropriate units** from the Units catalog
    3. **Use DECIMAL data type** for most measurements (temperature, humidity, pressure)
    4. **Document physical constraints** in the description field
    
    Example:
    ```json
    {
      "name": "SOIL_TEMPERATURE",
      "description": "Soil temperature at 10cm depth",
      "categoryId": 1,
      "defaultUnitId": 1,  // °C
      "dataType": "DECIMAL",
      "minExpectedValue": -10.0,
      "maxExpectedValue": 50.0
    }
    ```
  </Accordion>
  
  <Accordion title="Actuator Types">
    When creating actuator types:
    
    1. **Choose correct control type** (BINARY, CONTINUOUS, MULTI_STATE)
    2. **For CONTINUOUS**, always set min/max values (typically 0-100)
    3. **For MULTI_STATE**, define states in actuator_states catalog first
    4. **Document control behavior** in the description
    
    Example:
    ```json
    {
      "name": "NUTRIENT_PUMP",
      "description": "Variable speed nutrient injection pump (0-100% flow rate)",
      "categoryId": 2,
      "defaultUnitId": 2,  // %
      "controlType": "CONTINUOUS",
      "dataType": "INTEGER",
      "minExpectedValue": 0,
      "maxExpectedValue": 100
    }
    ```
  </Accordion>
  
  <Accordion title="Actuator States">
    When defining actuator states:
    
    1. **Use consistent naming** (uppercase: ON, OFF, AUTO, MANUAL)
    2. **Set isOperational correctly** (true = actuator functioning, false = error/stopped)
    3. **Choose intuitive colors**: Green (operational), Red (error), Gray (off), Orange (transitional)
    4. **Order by priority**: Critical states (ERROR) should have high displayOrder
    
    Example:
    ```json
    {
      "name": "STARTING",
      "description": "Actuator is starting up",
      "isOperational": true,
      "displayOrder": 4,
      "color": "#FFA500"  // Orange for transitional state
    }
    ```
  </Accordion>
  
  <Accordion title="Soft Deletes">
    - **Prefer deactivation** over deletion to preserve historical device configurations
    - **Only delete** truly unused custom types with zero associated devices
    - **Document deprecation** in description before deactivating
    - **Use activeOnly=true** in queries to filter out deprecated types
  </Accordion>
</AccordionGroup>
